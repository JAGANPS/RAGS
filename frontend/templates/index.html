Standard RAG (Vector-Based)

Traditional Retrieval-Augmented Generation using embedding similarity

How It Works
┌─────────────┐ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ Documents │───▶│ Chunking │───▶│ Embedding │───▶│ ChromaDB │ └─────────────┘ └─────────────┘ └─────────────┘ └─────────────┘ │ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ │ Response │◀───│ Claude │◀───│ Top-K │◀─────────┘ └─────────────┘ └─────────────┘ │ Retrieval │ └─────────────┘
Key Components
  • Document Chunking: Splits documents into 512-token overlapping segments
  • Embeddings: Uses sentence-transformers/all-MiniLM-L6-v2 (384 dimensions)
  • Vector Store: ChromaDB with cosine similarity search
  • Retrieval: Returns top-K most similar chunks
Performance Metrics
Speed: ~500ms
Accuracy: Good
Best For: Semantic Search
Strengths
  • Fast retrieval with pre-computed embeddings
  • Works well for semantic similarity queries
  • Proven, well-understood approach
  • Low latency for simple queries
Limitations
  • Similarity ≠ Relevance (can retrieve semantically similar but irrelevant content)
  • Struggles with structured document navigation
  • Cannot handle cross-references ("see Appendix G")
  • Fixed chunk boundaries may split important context

Vectorless RAG (Reasoning-Based)

PageIndex-inspired approach using hierarchical document structure

How It Works
┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ Documents │───▶│ Build ToC │───▶│ JSON Tree │ └─────────────┘ │ Tree │ │ Index │ └─────────────┘ └─────────────┘ │ ┌──────────────────────────────────┘ ▼ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ LLM Reads │───▶│ Reasons │───▶│ Navigates │ │ ToC │ │ "Where │ │ to Section │ └─────────────┘ │ to look?" │ └─────────────┘ └─────────────┘ │ ▼ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ Response │◀───│ Claude │◀───│ Extracts │ └─────────────┘ └─────────────┘ │ Content │ └─────────────┘
Key Innovation: No Embeddings Required

Instead of converting documents to vectors, Vectorless RAG builds a hierarchical Table of Contents (ToC) structure. The LLM then reasons about which sections are relevant, mimicking how a human expert would navigate a document.

Performance Metrics
Speed: ~2-3s
Accuracy: 98.7% (FinanceBench)
Best For: Structured Docs
Strengths
  • Preserves document structure and context
  • Handles cross-references naturally
  • 98.7% accuracy on financial document benchmarks
  • Explainable retrieval (shows reasoning)
  • No embedding storage overhead
Limitations
  • Higher latency due to LLM reasoning steps
  • Requires documents with clear structure
  • More API calls (higher cost)
  • Less effective for unstructured content

Agentic RAG (Multi-Agent)

Intelligent orchestration with planning, routing, synthesis, and reflection

How It Works
┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ Query │───▶│ ROUTER │───▶│ Complexity │ └─────────────┘ │ AGENT │ │ Analysis │ └─────────────┘ └─────────────┘ │ │ ┌───────────────┼───────────────┐ │ ▼ ▼ ▼ │ ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ Standard │ │ Vectorless │ │ Both │◀┘ │ RAG │ │ RAG │ │ (Hybrid) │ └─────────────┘ └─────────────┘ └─────────────┘ │ │ │ └───────────────┼───────────────┘ ▼ ┌─────────────────┐ │ SYNTHESIS │ │ AGENT │ └─────────────────┘ │ ▼ ┌─────────────────┐ │ REFLECTION │ │ AGENT │ └─────────────────┘ │ ▼ ┌─────────────────┐ │ Response │ └─────────────────┘
Agent Roles
  • Router Agent: Analyzes query complexity, chooses retrieval strategy
  • Retrieval Agents: Execute Standard and/or Vectorless retrieval
  • Synthesis Agent: Combines results from multiple sources
  • Reflection Agent: Validates answer quality, identifies gaps
Performance Metrics
Speed: ~4-6s
Accuracy: Highest
Best For: Complex Queries
Strengths
  • Handles multi-hop reasoning queries
  • Self-correcting with reflection
  • Chooses optimal retrieval strategy per query
  • Can decompose complex queries into sub-queries
  • Highest overall accuracy
Limitations
  • Highest latency (multiple LLM calls)
  • Most expensive (API costs)
  • May be overkill for simple queries
  • More complex to debug

Agent Capabilities

🔌 Query Router

Analyzes incoming queries to determine complexity (simple/moderate/complex) and selects the optimal retrieval strategy (vector/reasoning/hybrid). Can decompose complex queries into sub-queries.

💡 Synthesis Agent

Combines information from multiple retrieval sources into a coherent, well-structured answer. Cites sources and indicates confidence levels.

🔍 Reflection Agent

Reviews generated answers for accuracy, completeness, and relevance. Identifies gaps and suggests improvements. Can trigger additional retrieval if needed.

⚙ Powered by Agno Framework

Built on the Agno agent framework with AWS Bedrock integration. Uses Claude Sonnet 4 (us.anthropic.claude-sonnet-4-6) for all agent operations.

Decision Logic Example

Query: "What are the SAM best practices for audit preparation?" Router Analysis: complexity: "moderate" strategy: "hybrid" reasoning: "Query spans multiple concepts (SAM + audit + best practices). Hybrid approach will get semantic matches AND structured navigation." sub_queries: [] requires_iteration: false Retrieval: - Standard RAG: 3 chunks (relevance: 0.87, 0.82, 0.79) - Vectorless RAG: 2 sections (audit_checklist.txt, sam_best_practices.txt) Synthesis: Combined 5 sources into comprehensive answer Reflection: quality_score: 0.92 is_complete: true gaps_identified: []
